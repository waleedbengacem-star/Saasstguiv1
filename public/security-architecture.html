Section 01

Security Architecture Overview

The Holiday Homes SaaS platform implements a defense-in-depth security model with multiple overlapping layers. Each layer provides independent protection — if one fails, subsequent layers prevent unauthorized access.

Defense-in-Depth Stack

Layer 1 — Cloudflare Edge (SSL/TLS, DDoS, WAF) Network
Layer 2 — Next.js Middleware (Route Guard + Cookie Check) Gateway
Layer 3 — Server-Side Session Verification (DB Stateful) Application
Layer 4 — RBAC Permission Enforcement Authorization
Layer 5 — PostgreSQL Row-Level Security (Tenant Isolation) Database

Password Storage

// password.ts — bcryptjs with 10 salt rounds
import bcrypt from 'bcryptjs';

const SALT_ROUNDS = 10;

export async function hashPassword(password: string): Promise<string> {
  return bcrypt.hash(password, SALT_ROUNDS);
}

export async function verifyPassword(password: string, hash: string): Promise<boolean> {
  return bcrypt.compare(password, hash);
}
✓
Why bcrypt? — Adaptive cost function makes brute-force attacks computationally expensive. 10 salt rounds ≈ ~100ms per hash, sufficient for production use while resistant to GPU attacks.

Security Decisions

  • No plaintext password storage — only bcrypt hashes stored in the passwordHash column.
  • Constant-time comparison — bcrypt.compare() is inherently timing-safe.
  • Generic error messages — login returns "Invalid email or password" regardless of whether the email or password is wrong (prevents user enumeration).
  • Account deactivation check — user.isActive is verified before password comparison.

Section 04

Session Management

Stateful, server-side sessions stored in PostgreSQL. No JWTs — tokens are opaque random strings whose SHA-256 hashes are stored in the database.

Session Token Lifecycle

Generate 32 Random Bytes
→
Hex Encode (64 chars)
→
SHA-256 Hash
→
Store Hash in DB
→
Send Plaintext via Cookie
// session.ts
function generateToken(): string {
  return crypto.randomBytes(32).toString('hex');  // 256-bit entropy
}

function hashToken(token: string): string {
  return crypto.createHash('sha256').update(token).digest('hex');
}

Session Database Record

FieldTypePurpose
tokenHashString (unique)SHA-256 of the plaintext token — database never sees the plaintext
userIdUUID (FK)The authenticated user
organizationIdUUID (FK, nullable)Active tenant context for this session
ipAddressinetClient IP at session creation (audit/anomaly detection)
userAgentStringBrowser/device fingerprint
expiresAtTimestamptz7-day TTL — expired sessions are auto-revoked
⚠
Why hash session tokens? — If the database is compromised (SQL injection, backup leak), attackers cannot reconstruct valid session cookies from the stored hashes. This follows the "tokens as secrets" pattern from OWASP Session Management Guidelines.

Session Verification on Every Request

// Every protected API route / server component calls:
const session = await verifySession(sessionToken);

// Verification checks:
// 1. Hash the cookie token → lookup in sessions table
// 2. If not found → return null (invalid token)
// 3. If expired → delete session row, return null
// 4. Load user + org + role + permissions → return full context

Session Revocation

  • Logout: Session row is deleted from database + cookie is expired (set to epoch 0)
  • Expired: Sessions past their 7-day TTL are deleted on next verification attempt
  • Deactivated User: user.isActive = false prevents login; existing sessions continue until TTL but role check returns guest

Permission Enforcement Code

// permissions.ts — Server-side enforcement
export async function hasPermission(requiredPermission: string): Promise<boolean> {
  const permissions = getPermissionsFromSession();

  // Wildcard overrides for admins
  if (permissions.includes('*') || permissions.includes('admin.*')) {
    return true;
  }

  return permissions.includes(requiredPermission);
}

// Usage in API routes:
await requirePermission('accounting.manage_ledger');
// Throws 403 Forbidden if user lacks this permission

Permission Format

  • * — Wildcard: full access to everything (Super Admin / Org Admin)
  • properties.create — Resource-level: {resource}.{action}
  • owner_portal.view_statements — Portal-scoped: limits access to portal features only

Every data record belongs to an Organization. Isolation is enforced at the database level using PostgreSQL Row-Level Security, making cross-tenant data leakage impossible even if application code has bugs.

Section 07

Row-Level Security (RLS)

PostgreSQL-native row-level security enforces tenant isolation at the database engine level. RLS policies are transparent to the application — queries automatically return only rows belonging to the current organization.

RLS Policy Definition

-- rls.sql — Applied to all tenant tables

-- Helper functions read session context variables
CREATE FUNCTION get_current_org_id() RETURNS uuid AS $$
  SELECT NULLIF(current_setting('app.current_org_id', true), '')::uuid;
$$ LANGUAGE sql STABLE;

-- Example: Properties table policy
CREATE POLICY properties_isolation ON "properties"
  FOR ALL TO public
  USING ("organizationId" = get_current_org_id())
  WITH CHECK ("organizationId" = get_current_org_id());

Runtime Context Injection

// client.ts — Sets tenant context before every query
export async function runInTenantContext<T>(orgId, userId, fn) {
  return await prisma.$transaction(async (tx) => {
    // SET LOCAL scopes these variables to this transaction ONLY
    await tx.$executeRawUnsafe(
      `SET LOCAL app.current_org_id = '${orgId}';`
    );
    await tx.$executeRawUnsafe(
      `SET LOCAL app.current_user_id = '${userId}';`
    );
    return await fn(tx);
  });
}
✓
SET LOCAL guarantees: Session variables are scoped to the transaction only. They are automatically discarded when the transaction ends, preventing context leakage between concurrent requests.

Cookie & Transport Security

Session cookies are configured with defense-in-depth flags to prevent common web attacks.

Cookie Configuration

response.cookies.set('session_id', sessionToken, {
  httpOnly: true,      // Cannot be read by JavaScript (XSS protection)
  secure:   true,      // Only sent over HTTPS (production)
  sameSite: 'strict',  // Not sent with cross-origin requests (CSRF protection)
  path:     '/',       // Available to all routes
  expires:  expiresAt, // 7-day expiration
});
