Critical Findings Summary

The audit revealed severe flaws in both the inactive accounting utilities and the active owner-portal API route. Key problems include: hardcoded currency conversion rates which assume all non-USD currencies match AED 1:1, database ownership model mismatches bypassing fractional/contract-based property ownership table, in-memory aggregation of bookings without date boundaries presenting severe performance/scalability threat, and double-entry rounding math that can create unbalanced base-currency ledger transactions.

Recommendations & Action Plan

Suggested fixes to resolve findings
  • Implement Multi-Currency Support

    Refactor route.ts to fetch real-time or daily currency exchange rates from a configuration table or external integration, instead of hardcoding USD to AED as 3.6725 and defaulting all other currencies to 1.0.

  • Correct Property Ownership Lookup

    Modify the property query filter in route.ts to search for matches in the OwnerProperty join table rather than only relying on the direct Property.ownerContactId relation, ensuring fractional owners are correctly mapped.

  • Add Date Boundaries & Pagination

    Introduce startDate and endDate query parameters in the GET /api/owner-portal/revenue endpoint. Enforce database-level date range filters on bookings to restrict the queried volume, optimizing latency and avoiding timeouts.

  • Clean Up Dead Code & Obsolete Modules

    Remove the unused journal.ts file if it is obsolete. If double-entry bookkeeping will be implemented in the future, rewrite the base-currency conversion code to balance debits and credits on a journal level instead of rounding each line separately. Also, replace // @ts-nocheck with strict TypeScript type signatures and remove the unused prisma client import from route.ts.
