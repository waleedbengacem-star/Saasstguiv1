🔥 Critical Logical Failures

The following problems represent major logical bugs that directly impact the application's runtime capabilities or stability.

Bug #1: Invite Tokens Expire in 10 Minutes Instead of 7 Days Critical
Files: api/settings/team/route.ts • api/settings/team/resend-invite/route.ts

In the team invitation flow, invite tokens are configured to expire in exactly 10 minutes. However, the accompanying developer comments state that invitations should last 7 days.

Since email delivery and user response times often exceed 10 minutes, the vast majority of invited team members will encounter invalid/expired token errors when attempting to accept invites, blocking workspace onboarding. 

// src/app/api/settings/team/route.ts
// Line 87: Generate secure invite token (expires in 7 days)
const inviteToken  = crypto.randomBytes(32).toString('hex');
const inviteExpiry = new Date(Date.now() + 10 * 60 * 1000); // 10 minutes (bug)
Bug #2: Temporal Dead Zone (TDZ) React State Access Critical
File: dashboard/settings/page.tsx (Lines 173 - 315)

The primary state loading mechanism (useEffect) is declared at line 173, but the state setter variables it calls (such as setUplistingApiKey, setDsClientId, setClaudeApiKey) are declared much further down in the file (line 646).

In JavaScript, const variable declarations are not hoisted. Referencing these state setters textually above their declarations triggers Temporal Dead Zone (TDZ) violations and ESLint errors. This breaks standard React initialization practices and can cause compile failures or unpredictable state assignment issues depending on strict compiler configs. 

// src/app/dashboard/settings/page.tsx
// Line 173: Loading effect
useEffect(() => {
  ...
  setUplistingApiKey(dbSettings.pms_uplisting_api_key ...); // accessed before declaration
}, [...]);

...
// Line 646: State declarations
const [uplistingApiKey, setUplistingApiKey] = useState('');

🔗 API Architecture & Redundancy Issues

A review of the 12 route handlers under src/app/api/settings/ highlighted structural shortcuts and duplicated utility code.

Issue #1: Code Duplication of resolveLogoUrl()

The utility function resolveLogoUrl, which resolves relative logo paths to public absolute URLs for email templating, is duplicated verbatim across two API route files:

  • src/app/api/settings/team/route.ts (Lines 14-25)
  • src/app/api/settings/team/resend-invite/route.ts (Lines 11-20)

This violates the DRY (Don't Repeat Yourself) principle. It should be refactored into a shared utility file (e.g., @/lib/utils/url.ts).

Issue #2: Systematic Suppression of TypeScript Errors

Nine of the scanned API route files begin with the // @ts-nocheck directive, completely disabling TypeScript type validation for those files.

This bypasses type checking and permits unsafe variable assignments, missing property references (e.g., typing objects as any), and potential null-pointer crashes. Removing @ts-nocheck and resolving the type definitions is highly recommended.
